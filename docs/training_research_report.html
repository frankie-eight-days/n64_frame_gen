2. Architecture Specifics -- G-Buffer Conditioned Upscalers

2.1 Reference Architectures from the Literature

NSRD (Neural SR with Radiance Demodulation) CVPR 2024
Parameters
1.61M
Temporal
ConvLSTM
Reconstruction
U-shaped + RCAB
Training Time
~24 hrs/scene

Architecture: Radiance demodulation branch separates lighting from materials. Reliable warping module with gated convolution feeds warped lighting + motion masks into a ConvLSTM, followed by a U-shaped reconstruction module with residual channel attention blocks (RCAB). Uses pixel-unshuffle for HR-to-LR mapping.

Inputs: LR color, depth, normals, motion vectors, material G-buffers.

Loss: L_final = 1.0 * Smooth-L1 + 1.0 * (1 - SSIM) + 1.0 * L_temporal (motion-masked)

Training: 200 epochs, batch 8, 96x96 LR patches, LR 5e-4 (halved every 100 epochs), Adam, RTX 3090.

Source: Li et al., arXiv 2308.06699

Efficient Neural Supersampling (QRISP dataset) ICCV 2023

Conv Layers

1.08 / 1.53 / 3.52 ms

Architecture: 3x3 Conv-ReLU blocks with depth-to-space upsampling. Warping module with jitter compensation and depth-informed dilation for temporal re-projection. First/last conv kernels predicted by MLP conditioned on jitter offset (jitter-conditioned convolutions).

Inputs: Color C_t, depth D_t, jitter offset J_t, previous output Y_{t-1}, previous features f_{t-1}.

Loss: L1 (primary). Also tested weighted SSIM + VGG perceptual loss.

Training: 500K iterations, batch 8 clips x 16 frames, 264x264 HR patches, LR 1e-4 (halved at 200K/400K), Adam.

Dataset: QRISP -- 7,260 training frames from 10 scenes, 1,500 test frames from 3 scenes, 270p-1080p, with color/depth/motion vectors at each resolution.

Architecture: Four-level UNet backbone with 3 encoder/decoder stages + skip connections. Outputs per-pixel: 4x4 filter kernel, temporal accumulation coefficients, hidden state tensor (recurrent feedback). Input tensor: 12 channels (color, motion vectors, depth, luma derivative, disocclusion mask, reprojected hidden features).

Training: PyTorch + Adam + cosine annealing. Sequences of ~100 frames at 540p (1 spp) paired with 1080p ground truth (16 spp). Spatiotemporal loss penalizing both spatial fidelity and temporal consistency.

Source: Arm Developer Blog, HuggingFace

Architecture: Reparameterized conv blocks (3 convs collapsed to 1 at inference) + PixelUnshuffle for spatial downsampling + lightweight ConvLSTM for temporal aggregation. Without reparameterization, params rise from 138K to 298K. Without ConvLSTM, params drop to 65K but PSNR drops ~5 dB.

Key insight: Operates on encoded game frames (not raw renders). Bandwidth reduction of 35-49% for cloud gaming.

2.4 Temporal Component: ConvLSTM vs Feature Warping

ConvLSTM (used by NSRD, GameSR)

  • Maintains hidden state across frames
  • Learns to selectively accumulate temporal info
  • Adds ~50-200K params for 48-channel hidden state
  • Handles occlusion/disocclusion implicitly
  • NSRD: "effectively utilizes state of intermediate features to obtain more spatial-temporal correlation"
  • GameSR: removing ConvLSTM drops PSNR by ~5 dB

Feature Warping (used by QRISP, Arm NSS, Meta)

  • Warp previous frame features using motion vectors
  • Explicit re-projection is geometrically accurate
  • Needs disocclusion masks for ghosting prevention
  • Lower compute cost than ConvLSTM
  • QRISP: depth-informed dilation + jitter compensation
  • NSRD uses both: warp then ConvLSTM
Recommendation

Use feature warping with motion vectors (available from N64 emulator) as the primary temporal mechanism, with a lightweight ConvLSTM (48 hidden channels) for residual temporal refinement. This mirrors NSRD's approach and leverages the N64's readily-available motion vector data from the RDP pipeline.

2.5 Loss Functions

Loss Component Weight Purpose Used By
L1 / Smooth-L1 1.0 Pixel-level fidelity, robust to outliers NSRD, QRISP, SPAN, RLFN, GameSR
SSIM Loss 1.0 Structural similarity, perceptual quality NSRD
VGG Perceptual (relu3_3) 0.01-0.1 Feature-level texture fidelity QRISP (tested), Real-ESRGAN
Temporal Consistency 1.0 Penalize flickering between warped frames NSRD (motion-masked), Arm NSS
Adversarial (GAN) 1e-3 Sharp, realistic textures (adds training complexity) Real-ESRGAN, SRGAN
Recommended Loss for N64 Upscaler

L_total = 1.0 * L1 + 1.0 * (1 - SSIM) + 1.0 * L_temporal

This matches NSRD's proven formula. Add VGG perceptual loss (weight 0.05) if textures look too smooth. Avoid GAN loss for initial training -- it complicates convergence and is unnecessary for N64's low-frequency content.

3.3 Generating N64 Training Data

Option A: paraLLEl-RDP (Recommended)

  • Native resolution: 320x240 (256x224 for some games)
  • Upscaled output: 2x, 4x, or 8x internal resolution via Vulkan
  • At 4x: 1280x960 output -- our exact target
  • At 8x: 2560x1920 output -- even higher quality GT
  • Fully emulates N64 RDP with VI Interface
  • Equivalent of 8x MSAA at highest setting
  • How to extract data: Run RetroArch with paraLLEl-RDP core. Capture native-res frames + hi-res frames simultaneously via frame dumping. Z-buffer is accessible through the RDP pipeline.
  • Available in RetroArch (Mupen64Plus-Next core)

Option B: RT64

  • N64 graphics renderer with enhancements
  • Runs two renderers simultaneously: one at native 240p, one at arbitrary higher resolution
  • Replays all draw calls at higher resolution
  • Can upscale framebuffer effects
  • Better for native SM64 ports (SM64 PC port, SM64 Render96)
  • Advantage: Direct access to draw calls = can extract depth, normals, and motion vectors per-frame
  • Disadvantage: Game-specific integration needed
Recommended Data Pipeline
  1. Use paraLLEl-RDP at 4x (1280x960) as HR ground truth
  2. Simultaneously capture native 320x240 as LR input
  3. Extract Z-buffer from RDP pipeline as depth map
  4. Compute motion vectors from frame-to-frame depth + camera reprojection
  5. Compute surface normals from depth gradients (Sobel filter on Z-buffer)
  6. Record 5,000-10,000 frames across diverse game scenes (SM64 levels, OoT areas, etc.)
  7. Resolution pair: 320x240 -> 1280x960 (4x upscale)

GTX 4060 Estimated Training Speed

Calculation

The GTX 4060 has approximately 50-60% of the throughput of an RTX 3090 for training workloads (15 TFLOPS FP32 vs 36 TFLOPS, but 4th-gen tensor cores partially compensate with FP16). For a 1.5M parameter model:

  • NSRD (1.61M params, 200 epochs, RTX 3090) = 24 hours
  • GTX 4060 equivalent: ~36-48 hours for similar training run
  • For a full 500K-1M iteration run: 3-7 days

Estimated iteration speed on GTX 4060 (batch 16, 96x96 patches, FP16):

  • ~2-4 iterations/second
  • 500K iterations: ~35-70 hours (1.5-3 days)
  • 1M iterations: ~70-140 hours (3-6 days)

6. Training Frameworks & Memory Optimization

6.1 BasicSR

BasicSR (XPixelGroup) Recommended

What it is: Open-source PyTorch toolbox for image/video restoration. Used by Real-ESRGAN, SwinIR, EDVR, BasicVSR, SPAN, SAFMN, RLFN, and most NTIRE challenge winners.

How to use with custom architectures:

  1. Define your model in basicsr/archs/your_model_arch.py
  2. Register it with @ARCH_REGISTRY.register()
  3. Create a YAML config specifying architecture, dataset, training params
  4. Run: python basicsr/train.py -opt options/train/your_config.yml

Key features:

  • Registry pattern for models, losses, datasets (plug-and-play)
  • Built-in: L1, L2, perceptual/VGG, GAN losses
  • Built-in: DIV2K, LMDB dataset loaders
  • DDP multi-GPU training out of the box
  • Logging, checkpointing, validation metrics (PSNR, SSIM)
  • Easy to add custom datasets (just implement a DataLoader)

Source: GitHub

6.2 HuggingFace Accelerate

HuggingFace Accelerate Alternative

What it is: Lightweight wrapper for PyTorch training that handles distributed training, mixed precision, and gradient accumulation with minimal code changes.

Key features for this project:

  • Mixed precision: fp16, bf16, fp8 with one flag
  • Gradient accumulation: simulate larger batch sizes
  • Single GPU, multi-GPU, TPU support without code changes
  • Integrates with DeepSpeed and FSDP
  • Pro: More flexible for custom training loops than BasicSR
  • Con: No built-in SR-specific losses or metrics; you write more boilerplate

Source: GitHub

7.3 Comparison to Current SD-Turbo Pipeline

Metric Current (SD-Turbo) Proposed (CNN + G-buffer) Improvement
FPS ~39 FPS 120-500+ FPS 3-13x faster
Inference VRAM ~4 GB ~65-115 MB 35-60x less

Model Size ~2-4 GB (SD weights) ~3-6 MB 500-1300x smaller

Quality Tradeoff Note

The CNN approach will produce a faithful 4x upscale -- sharper edges, proper geometry from G-buffers, stable temporally. But it will not "hallucinate" new texture details the way SD-Turbo does. The output will look like a very clean, sharp version of the N64 game rather than a reimagined HD version. For an N64 game running at 320x240, the CNN approach is arguably more appropriate -- it preserves the artistic intent while removing aliasing and blur.

8. Recommended Training Plan

Phase 1: Data Collection (1-2 days)

  1. Set up RetroArch with paraLLEl-RDP core
  2. Configure dual capture: native 320x240 + 4x upscaled 1280x960
  3. Enable Z-buffer output from RDP pipeline
  4. Record 5,000-10,000 frames across 5-10 diverse SM64 levels
  5. Compute motion vectors from frame-pair depth reprojection
  6. Compute surface normals from depth gradients
  7. Convert dataset to LMDB format for fast training I/O

Phase 2: Model Implementation (1 day)

  1. Implement dual-branch architecture in PyTorch (see Section 2.3)
  2. Register in BasicSR or set up custom training loop with HF Accelerate
  3. Implement loss functions: L1 + SSIM + temporal consistency
  4. Write YAML training config
  5. Sanity check: overfit on 10 frames to verify architecture works

Phase 3: Training (3-7 days local OR 1 day cloud)

Option A: Local (GTX 4060, free)

  • 500K iterations, batch 16, 96x96 patches, FP16
  • Estimated: 3-5 days
  • VRAM usage: ~3-4 GB

Option B: Cloud (Vast.ai RTX 4090, ~$4-6)

  • 500K iterations, batch 64, 96x96 patches, FP16
  • Estimated: 12-18 hours
  • Upload dataset (~2-5 GB), run training, download checkpoint

Option C: Cloud (Vast.ai A100, ~$8-10)

  • 1M iterations, batch 64, larger patches
  • Estimated: 18-24 hours
  • Best quality, reasonable cost

Phase 4: Optimization & Deployment (1 day)

  1. Export to ONNX: torch.onnx.export(model, ...)
  2. Convert to TensorRT FP16: trtexec --onnx=model.onnx --fp16
  3. Optionally quantize to INT8 for maximum speed
  4. Integrate into existing N64 DLSS pipeline, replacing SD-Turbo
  5. Benchmark: target 120+ FPS at 320x240 -> 1280x960
Total Timeline: 6-11 days (local) or 4-5 days (with cloud training)

Total cost if using cloud: $4-25 depending on GPU and iterations. This is a realistic, achievable project that would produce a dramatically faster and lighter upscaler than the current SD-Turbo pipeline.
