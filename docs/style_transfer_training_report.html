N64 Frame Generation: Style Transfer Training Guide

How to train a model that transforms N64 frames into Ghibli, realistic, watercolor, and other styles — using depth + normals from the emulator for structural preservation.

TL;DR

The goal: Replace our current SD-Turbo pipeline with a model specifically trained to transform N64 frames into various art styles, using depth and normals from the emulator so it preserves scene geometry perfectly.

The approach: Use a large model (SDXL) offline to generate 50K styled training targets, then train a small fast model to replicate those outputs given the N64 frame + G-buffer data.

Can you train on your GTX 4060? Yes — a feedforward style network trains in 6-12 hours per style. A GAN takes 3-5 days. Both run overnight while you're away.

Or use cloud? A100 on Vast.ai costs ~$0.50/hr. Total training: $5-30 depending on approach.

Result: 60-500 FPS inference (vs 39 FPS now), using <1GB VRAM (vs 4GB now), with better structural preservation thanks to G-buffer conditioning.

1. The Core Idea: Distillation

The key insight is that we don't need a 860M parameter diffusion model running at inference time. Instead:

SDXL + ControlNet
(offline, slow, high quality)
→
50K styled frames
(training targets)
→
Train small model
(2-50M params)
→
Real-time inference
(60-500 FPS)

This is how the best results are achieved in practice: large models generate gold-standard outputs, then a small model learns to replicate them. The small model can also accept additional inputs (depth, normals) that the large model didn't have — giving it structural advantages at a fraction of the cost.

3. Option A: Feedforward Style Network (Recommended Start)

Why Start Here

This is the fastest to train, fastest at inference, and simplest to implement. You train one small model per art style. Each model is ~8MB and runs at 200+ FPS. You can train all styles overnight on your GTX 4060 — no cloud needed.

Architecture

Based on Johnson et al. "Perceptual Losses for Real-Time Style Transfer" (2016), modified for 7-channel input:

Input (7ch: RGB + Depth + Normals, 320×240)
  ↓
Conv 9×9, 32 filters, ReLU, InstanceNorm
Conv 3×3 stride 2, 64 filters (downsample)
Conv 3×3 stride 2, 128 filters (downsample)
  ↓
5× Residual Blocks (128 filters each)
  ↓
ConvTranspose 3×3 stride 2, 64 filters (upsample)
ConvTranspose 3×3 stride 2, 32 filters (upsample)
Conv 9×9, 3 filters, Tanh
  ↓
Output (3ch RGB, 320×240)

Total parameters: ~1.7M. Model size: ~7MB.

Loss Function

  • Perceptual loss (VGG-16 features at relu2_2 and relu4_3) — makes output look "natural"
  • Style loss (Gram matrices of VGG features from a reference style image) — drives artistic style
  • Pixel L1 loss against the SDXL-generated target — anchors the output
  • Temporal loss (L1 between warped previous output and current output) — optional, for video consistency

How Depth + Normals Help

The 7-channel input gives the network explicit 3D structure. Without it, the network only sees RGB pixels and has to guess where edges, surfaces, and objects are. With depth + normals:

  • The network knows which pixels are foreground vs background → better edge preservation
  • Normals tell it surface orientation → better shading and style application
  • Depth discontinuities signal object boundaries → cleaner object silhouettes

4. Option B: Compact GAN

When to Use This

If Option A's quality isn't sharp enough (feedforward nets can be slightly soft/blurry), upgrade to a GAN. The adversarial discriminator produces crisp, detailed outputs with better textures. Think of it as "Option A but with a quality boost at the cost of longer training."

Architecture

Based on Pix2PixHD / HyPER-GAN, adapted for multi-channel input:

Generator: U-Net encoder-decoder
  Input: 7 channels (RGB + depth + normals)
  Encoder: 4 downsampling blocks (64→128→256→512)
  Bottleneck: 6 residual blocks
  Decoder: 4 upsampling blocks with skip connections
  Output: 3 channels (RGB)
  Params: ~20-50M

Discriminator: Multi-scale PatchGAN
  3 discriminators at 1x, 0.5x, 0.25x scale
  Each: 4 conv layers → real/fake score per patch
  Params: ~5M total

Loss Function

  • Adversarial loss (hinge or LSGAN) — drives sharpness and realism
  • L1 pixel loss against SDXL target — structural fidelity (weight: 10-100)
  • VGG perceptual loss — naturalness (weight: 1-10)
  • LPIPS loss — perceptual similarity (optional, weight: 1)
  • Feature matching loss — match discriminator intermediate features (stabilizes training)

5. Option C: Fine-Tuned Diffusion

When to Use This

If you want prompt-controllable artistic generation — change the style at runtime by changing the text prompt, not by swapping models. This is the most flexible but slowest option. Similar speed to your current pipeline, but with much better structural preservation from G-buffer inputs.

How It Works

Expand SD-Turbo's UNet from 4 input channels (latent) to 8 (latent + depth + normals). Zero-initialize the new channels so it starts from pretrained behavior, then LoRA fine-tune.

# Expand UNet input channels
unet = UNet2DConditionModel.from_pretrained("stabilityai/sd-turbo", subfolder="unet")
old_conv = unet.conv_in  # shape: [320, 4, 3, 3]

new_conv = torch.nn.Conv2d(8, 320, kernel_size=3, padding=1)
new_conv.weight.data[:, :4] = old_conv.weight.data  # preserve pretrained weights
new_conv.weight.data[:, 4:] = 0  # zero-init new channels
new_conv.bias.data = old_conv.bias.data
unet.conv_in = new_conv

# Apply LoRA to attention layers
from peft import LoraConfig, get_peft_model
lora_config = LoraConfig(r=32, target_modules=["to_q", "to_k", "to_v", "to_out.0"])
unet = get_peft_model(unet, lora_config)
# conv_in is fully trainable (not LoRA) — it needs to learn new channel semantics

6. Training Data: Where Do Styled Targets Come From?

This is the critical question. For all three options, you need paired data: (N64 frame + depth + normals) → styled frame.

Step 1: Capture N64 Frames + G-Buffers

Instrument your emulator to auto-save during gameplay:

  • RGB frame (320×240) — already captured by your frontend
  • Depth buffer (320×240 float32) — already extracted from RDRAM
  • Normals (320×240×3 float32) — compute from depth gradients (3 lines of code)

Play through diverse content: multiple games, levels, lighting conditions, camera angles. Target: 50K-100K frames.

Step 2: Generate Styled Targets with SDXL (Offline)

For each frame, generate targets using prompts like:

  • Ghibli: "studio ghibli anime style, lush detailed background, soft lighting"
  • Realistic: "photorealistic modern 3D game, unreal engine 5, ray traced lighting"
  • Watercolor: "watercolor painting, soft edges, flowing colors, artistic"

Use ControlNet-Depth with your extracted Z-buffer to preserve scene geometry in the styled output.

Alternative: Unpaired Training (No Target Generation)

If you don't want to generate paired targets, CycleGAN-Turbo or CUT can learn style transfer from two unpaired collections:

  • Set A: N64 screenshots (50K+)
  • Set B: Ghibli artwork / modern game screenshots / watercolor paintings (50K+)

Quality is lower than paired training, but requires zero offline generation.

9. The Killer Advantage: Why G-Buffers Change Everything

Your current pipeline has the "dead zone" problem: low denoise strength preserves the scene but barely changes it; high denoise strength applies strong style but destroys the scene. There's a narrow sweet spot that limits quality.

With depth + normals as explicit inputs, the model knows the 3D structure independently of the RGB pixels. This means:

  • The model can completely restyle the RGB pixels (strong artistic effect)
  • While perfectly preserving geometry (depth tells it where things are, normals tell it surface orientation)
  • No more trading style intensity vs scene preservation — you get both

This is fundamentally why DLSS works so well: it has access to the scene structure separate from the pixel colors.

10. Recommended Plan

Phase 0: Compute Normals from Depth (30 minutes)

Add 3 lines of code to compute screen-space normals from the Z-buffer you already extract. This gives you the normals channel for free — no emulator modifications needed.

dzdx = depth[:, 2:] - depth[:, :-2]  # horizontal gradient
dzdy = depth[2:, :] - depth[:-2, :]  # vertical gradient
normals = normalize(cross(vec3(2, 0, dzdx), vec3(0, 2, dzdy)))

Phase 1: Capture Training Data (2-3 days)

Add auto-save to the emulator: every frame, save RGB + depth + normals to disk. Play through SM64, Zelda OoT, Mario Kart, etc. Aim for 50K diverse frames.

Phase 2: Generate Style Targets (1-3 days)

Batch-process captured frames through SD 1.5 + ControlNet-Depth locally (free, ~28 hrs for 50K) or SDXL on cloud ($28 for 50K). Generate targets for each desired style.

Phase 3: Train the Model (overnight to 5 days)

Start with Option A (feedforward, 6-12 hrs per style on your 4060). If quality isn't sufficient, upgrade to Option B (GAN, 3-5 days). Keep the existing SD-Turbo pipeline as a fallback for prompt-based creativity.
