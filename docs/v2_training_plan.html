N64 Frame Gen v2 — Training Plan

Prompt-controllable style transfer with G-buffer conditioning. Budget: ~$100.

Which Base Model?

Before diving into steps, the big question: is SD-Turbo still the right choice, or should we switch?

Model Base Steps Inference VRAM LoRA Ecosystem Pros Cons
SD-Turbo SD 2.1 1 ~3 GB Small (SD 2.x is less popular) Already working in our pipeline. Fastest (1 step). Smallest VRAM. SD 2.x ecosystem is tiny. Few community LoRAs. Quality ceiling is lower.
SD 1.5 + LCM-LoRA SD 1.5 2-4 ~3.5 GB ★★★★★ Massive Huge ecosystem — thousands of community style LoRAs work instantly. Best fine-tuning docs. Most ControlNet models. 2-4 steps instead of 1 (slightly slower). Needs LCM-LoRA for speed.
SDXL-Turbo SDXL 1-4 ~6-7 GB ★★★★ Large Best quality. Strong ecosystem. Tight on 8GB GPUs. Other people with 8GB cards can't run it comfortably alongside the emulator.
SDXL-Lightning SDXL 1-4 ~6-7 GB ★★★★ Large Higher quality than SDXL-Turbo at 2-4 steps. Same VRAM problem as SDXL-Turbo.

Recommendation: SD 1.5 + LCM-LoRA

Why switch from SD-Turbo:

  • Ecosystem: SD 1.5 has thousands of community style LoRAs (Ghibli, anime, realistic, pixel art, etc.) that work out of the box. SD 2.x (what SD-Turbo is based on) has almost none.
  • ControlNet: The best ControlNet models (depth, canny, normal) are all SD 1.5. Our G-buffer conditioning integrates naturally.
  • Shareability: Anyone with a 6-8GB GPU can run SD 1.5. SDXL cuts out half the audience.
  • Fine-tuning: SD 1.5 has the most documentation, tutorials, and tooling for LoRA training. HuggingFace, Kohya, every guide is SD 1.5 first.
  • LCM-LoRA: Drops inference to 2-4 steps with minimal quality loss. Combined with StreamDiffusion + TensorRT, this should hit 25-40 FPS — comparable to SD-Turbo's 39 FPS.

Tradeoff: 2-4 steps instead of 1. Slightly slower. But the ecosystem advantage is enormous — users can drop in any community LoRA and it just works.

SDXL is tempting but at 6-7GB inference VRAM, it leaves almost nothing for the emulator + display on an 8GB card. If you want this to be downloadable and usable by others, SD 1.5 is the right call.

The Steps

1

Compute Normals from Z-Buffer

Add code to derive screen-space normals from the depth buffer we already extract. Cross-product of depth gradients gives a per-pixel normal map.

Time: 30 minutes
Cost: $0
Where: Local

Three lines of numpy. We already have the Z-buffer — this is free data.

dzdx = z14[:, 2:] - z14[:, :-2]
dzdy = z14[2:, :] - z14[:-2, :]
normals = np.stack([-dzdx, -dzdy, np.ones_like(dzdx) * 2], axis=-1)
normals = normals / np.linalg.norm(normals, axis=-1, keepdims=True)
2

Build Data Capture Mode

Add an auto-save mode to the emulator: every Nth frame, write RGB + depth + normals to disk. Then play through diverse game content.

Time: 1-2 hrs coding + 2-3 days gameplay
Cost: $0
Where: Local

Target: ~50,000 frames from diverse scenes.

Games: SM64 (all levels), Zelda OoT (indoor/outdoor/dungeons/day/night), Mario Kart 64 (varied tracks), Banjo-Kazooie, etc.

Format: PNG for RGB, NPZ for depth + normals. ~1-2 KB per frame = ~50-100 GB total.

Tip: Save every 3rd-5th frame to avoid near-duplicate consecutive frames.

3

Generate Styled Training Targets (Cloud)

Batch-process every captured frame through SDXL + ControlNet-Depth on a cloud A100 to produce "gold standard" styled outputs for each desired style.

Time: ~14 hrs per style
Cost: ~$15 per style (Vast.ai A100)
Where: Cloud — A100 80GB

Pipeline per frame:

  1. Load captured N64 RGB frame
  2. Load corresponding depth map → feed to ControlNet-Depth
  3. Run SDXL at 30-50 steps, cfg 7.5, with style prompt
  4. Save output as training target

Styles to generate (3 styles = ~$45):

  • "studio ghibli anime style, lush detailed, soft lighting, beautiful background"
  • "photorealistic modern 3D game, unreal engine 5, ray traced, high detail"
  • "watercolor painting, soft edges, flowing colors, artistic, beautiful"

Why ControlNet-Depth matters: It forces SDXL to preserve the scene geometry while restyling. Without it, SDXL would hallucinate different layouts. With it, every styled target has the exact same spatial structure as the N64 original — perfect paired training data.

4

Set Up Training Pipeline

Write the training code. Expand SD 1.5's UNet input channels from 4→8, configure LoRA, build the dataloader.

Time: 1-2 days
Cost: $0
Where: Local (code only)

Framework: HuggingFace diffusers + peft (LoRA) + accelerate (mixed precision)

Key components:

  • Expand conv_in from 4→8 channels, zero-init new weights
  • LoRA rank 32 on all attention Q/K/V/out projections
  • Dataloader: loads (N64 frame, depth, normals, style prompt, SDXL target) tuples
  • VAE encodes both input frame and target to latent space
  • Depth + normals resized to latent resolution (40×30) and concatenated
  • Standard diffusion noise-prediction loss
  • Apply LCM-LoRA for 2-4 step inference after training
5

Train on Cloud

Upload data to cloud GPU, launch training, wait.

Time: 8-16 hours
Cost: ~$16-25
Where: Cloud — A100 80GB or H100

Training config:

  • LoRA rank 32, ~8-16M trainable parameters
  • Batch size 4-8 on A100 (much more room than 4060)
  • Mixed precision (bf16 on A100)
  • 30K-50K training steps
  • Learning rate: 1e-4 for conv_in, 1e-5 for LoRA layers

On A100 80GB: ~2-4 steps/second with batch 8 → 50K steps in ~4-7 hours. Add overhead for setup/data transfer → ~8-16 hours total rental time.

6

Export + Optimize

Convert trained model to TensorRT for real-time inference on your GTX 4060.

Time: 1-2 days
Cost: $0
Where: Local
  • Merge LoRA weights into base model
  • Apply LCM-LoRA for 2-4 step inference
  • Export modified UNet to ONNX → TensorRT FP16
  • Integrate into StreamDiffusion pipeline
  • Modify process_loop() to feed depth + normals alongside RGB

Expected result: 25-40 FPS with prompt-controllable style transfer + perfect structural preservation.

7

Test + Iterate

Try prompts, evaluate quality, refine if needed.

Time: Ongoing
Cost: ~$15 budget for reruns
Where: Local + Cloud as needed
  • If a specific style needs improvement → generate more targeted training data for that style, retrain (shorter run since you're refining)
  • Community LoRAs: test if existing SD 1.5 style LoRAs compose well with the G-buffer-trained LoRA
  • Tune LCM step count (2 vs 4) for speed/quality tradeoff
  • Consider publishing the trained LoRA + modified conv_in weights on HuggingFace for others to use
